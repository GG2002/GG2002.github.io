假设在无线网条件下，网络比较拥堵，你会选择 TCP 还是 UDP 通信

在无线网络条件下，如果网络比较拥堵，选择 TCP 还是 UDP 通信取决于你的应用场景和需求。

  • TCP（传输控制协议）：TCP 是一种面向连接的、可靠的传输层协议。它提供了确认机制、重传机制以及流量控制和拥塞控制，确保数据能够准确无误地到达目的地。在网络拥堵的情况下，TCP 会通过其内置的拥塞控制算法来降低发送速率，从而减少网络负担，试图避免进一步的拥堵。然而，在高丢包率或信号不稳定的无线网络环境中，TCP 的性能可能会受到影响，因为它可能频繁地进行重传操作，并且在检测到丢包时显著降低传输速度。

  • UDP（用户数据报协议）：与 TCP 不同，UDP 是一种无连接、不可靠的传输层协议。它不提供确认机制或重传机制，也没有内置的拥塞控制。这意味着 UDP 通常具有较低的延迟和开销，因为不需要等待确认即可发送下一个数据包。在某些实时应用中，如视频会议、在线游戏或 VoIP 等，即使网络条件不佳，也更倾向于使用 UDP，因为这些应用优先考虑传输的及时性而不是可靠性。对于这些场景来说，少量的数据丢失是可以接受的，重要的是要保持低延迟。

因此，如果你的应用对数据传输的可靠性要求较高，或者你需要确保所有数据都能被接收方正确接收，那么即使在网络拥堵情况下也应该选择 TCP。而如果你的应用可以容忍一定的数据丢失，并且需要尽可能低的延迟，比如实时音视频流媒体，则应选择 UDP。此外，现代的一些基于 UDP 的改进协议，如 QUIC，结合了部分 TCP 的优点并减少了 UDP 的缺点，也为开发者提供了更多的选择。

拥塞控制算法有哪些？

拥塞控制算法主要用于网络通信中，以确保数据能够高效、有序地传输，并避免因过多的数据同时传输而导致的网络过载问题。以下是一些常见的拥塞控制算法：

  1. TCP 拥塞控制算法：

    • 慢启动（Slow Start）：在连接开始时或网络出现丢包后，发送方以指数增长的方式增加拥塞窗口大小。
    • 拥塞避免（Congestion Avoidance）：当拥塞窗口达到某个阈值后，采用线性增长方式缓慢增加拥塞窗口大小。
    • 快速重传（Fast Retransmit）：一旦接收方检测到丢失了一个分组，它会立即发送多次重复的 ACK 给发送方，促使发送方在未等到超时的情况下就进行重传。
    • 快速恢复（Fast Recovery）：与快速重传配合使用，在检测到丢包时不立即进入慢启动状态，而是尝试通过调整拥塞窗口和阈值来快速恢复正常的数据传输。

  2. Reno 和 Tahoe：这些是 TCP 拥塞控制机制的具体实现版本，其中 Reno 对快速重传和快速恢复机制进行了改进。

  3. New Reno：进一步改进了 Reno，使得在多个分组丢失的情况下能更有效地恢复。

  4. CUBIC：一种广泛使用的 TCP 拥塞控制算法，旨在提供更好的带宽利用率，尤其是在高带宽 - 延迟产品网络环境中。

  5. BBR（Bottleneck Bandwidth and RTT）：由 Google 提出的一种新型拥塞控制算法，它试图找到并利用瓶颈带宽，从而最大化吞吐量，减少排队延迟。

  6. QUIC 协议中的拥塞控制：QUIC 是一个基于 UDP 的现代传输层协议，其拥塞控制机制借鉴了 TCP 的一些概念，但设计上更加灵活，支持更快的连接建立和更有效的拥塞控制。

以上就是一些常用的网络拥塞控制算法，每种算法都有其适用场景和特点。随着网络环境和技术的发展，新的算法也在不断被提出和发展。

1.首先会涉及操作系统内核协议栈的处理，就 linux 来说，如果 redis 服务在本机，写套接字这个过程会在到达协议栈传输层时，发现目的 ip 来自本机，然后返回。redis 读取请求，通过套接字把结果返回。

2.整个过程过程会有四次用户态和内核态的交互，其中两次还包括结果数据拷贝。

3.Redis 底层通过 epoll 函数来监测是否有请求到来，如果不想单核 CPU 占用率到 100%，那么 epoll 的 idle time 这个参数最低只能配置成 1ms，这意味着响应的平均耗时应该在 500us。
