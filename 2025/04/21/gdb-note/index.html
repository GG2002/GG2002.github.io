如何假装自己是 GDB 糕手

发表于2025-04-21|更新于2025-04-22
|字数总计:1.2k|阅读时长:4分钟|阅读量:

一、基础调试命令

命令 描述 示例
gdb [可执行文件] 直接加载可执行文件调试 gdb my_program
gdb -p <PID> / gdb attach <PID> 附加到正在运行的进程 gdb attach 1234
break <函数名> 在函数入口设断点 break main
break <文件名>:<行号> 在指定文件行设断点 break main.cpp:10
break if <条件> 条件断点 break 100 if cnt == 100
run [参数] 启动程序 run --test
continue (c) 继续执行到下一个断点 c
next (n) 单步执行（不进入函数） n
step (s) 单步执行（进入函数） s
print <变量名> 查看变量值，可以直接打印表达式 print x
bt / backtrace 打印当前堆栈 bt
delete <断点号> 删除断点 delete 1

二、多线程调试核心命令

命令 描述 示例
info threads 查看所有线程及状态 info threads
thread <线程ID> / t <ID> 切换到目标线程 thread 3
thread apply all bt 所有线程打印堆栈（排查死锁） thread apply all bt
set scheduler-locking on 锁定当前线程，其他线程暂停 set scheduler-locking on
break <函数名> thread <ID> 仅在指定线程生效的断点 break critical_func thread 2

三、Core Dump 分析

命令 描述 示例
gdb [可执行文件] core 加载 core 文件分析崩溃原因 gdb my_program core
bt full 打印完整堆栈（含变量值） bt full
info registers 查看寄存器状态（定位崩溃指令） info registers
x/<格式> <地址> 查看内存内容（如十六进制） x/10x 0x123456

四、内存泄漏检测

工具/命令 描述 示例
valgrind --tool=memcheck ... 检测内存泄漏 valgrind --leak-check=full ./a.out
valgrind --tool=massif ... 分析内存使用趋势 valgrind --massif ./a.out
x/<格式> <地址> 手动查看内存区域 x/100b 0x123456
p sizeof(...) 计算变量或结构体的内存占用 p sizeof(struct MyStruct)

五、面试装熟必杀技（附赠）

问题 1：调试过复杂多线程问题吗？
回答：
“上周我遇到线程 A 和 B 竞态写共享变量，直接gdb attach <PID>，用thread apply all bt拉出所有线程堆栈，发现两个线程在critical_section()同时阻塞。然后啪啪打了个条件断点break critical_section if cnt > 100，抓到线程 B 违规操作，定位到锁未释放的漏洞！”

问题 2：怎么分析 core dump？
回答：
“先用ulimit -c unlimited确保 core 文件生成，然后gdb your_exe core加载。用bt full直接看崩溃堆栈，发现是野指针*ptr = 7越界了。再结合info registers看寄存器状态，确认无效地址。最后用watch ptr设观察点，下次运行直接抓现行！”

问题 3：查内存泄漏？
回答：
“我一般用 Valgrind 的 massif 工具生成内存趋势图，但 GDB 也能玩出花！比如用x/1000x <地址>看内存块是否释放，或者写循环call free()强行清理（虽然不优雅但管用）。还有次用info malloc-history追踪到大对象未释放，揪出死循环的new操作！”

终极装熟话术：
“其实我最爱用的是set follow-fork-mode child，调试多进程时自动追踪子进程，配合set detach-on-fork off同时看父子进程状态，比 gdbserver 还丝滑。还有until命令，碰到讨厌的循环直接until 100跳到第 100 行，比 F5 快多了！”

附：GDB 黑话速成表

术语 翻译 作用说明
scheduler-locking 调度器锁定 让 GDB 只跑当前线程，其他线程暂停
inferior 附属进程 多进程调试时切换父/子进程
watchpoint 观察点 监控变量读写，比断点更狠
bt full 全量堆栈跟踪 打印堆栈时连局部变量都给你看
core file 核心转储文件 程序崩溃时的内存快照，救命神器

终极心法： 面试时多说"我一般会thread apply all bt先看看所有线程在干啥"，再补一句"然后set scheduler-locking on让线程乖乖排队"，立刻显得你掌控全场！

GDB 多线程调试小连招

我用过 gdb，比如调试正在运行的程序，先用 ps -ef 获取 pid，然后使用 gdb attach pid。

调试多线程，我会先 list 看看在哪行源码，或者在那个 C++ 文件，break 可以 xxx.cpp:30，在某个文件上第 30 行打个断点，然后 run 跑起来，info thread 左上角的星号是正在执行的线程，可以用 thread id 来切换线程，还可以用 set scheduler-locking 来设置只让一根线程执行。
